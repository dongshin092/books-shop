# books-shop Agent Team 오케스트레이션 지시서

> 이 문서를 Claude Code 세션에 처음 전달하면, Claude Code 는 **7번에 따라 `/agent-team`
> 슬래시 커맨드를 먼저 생성**합니다. 이후부터는 `/agent-team` 한 줄로 팀을 실행할 수 있습니다.
> 실행 시 팀장이 이 지시에 따라 기획자·프론트·백엔드 teammate를 스폰하고 작업을 조정합니다.

---

## 0. 사전 조건 (사람이 미리 준비)

- `settings.json` 의 env 에 아래 한 줄 추가 후 Claude Code 재시작:
  ```json
  { "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }
  ```
- 모델: 팀장(lead) Opus, teammate 들은 Sonnet 권장 (비용 절감)
- **환경: Windows + in-process 모드.** tmux 불필요.
  teammate 들은 메인 터미널의 agent 패널에 표시되며, 위/아래 화살표로 선택 후 Enter 로 확인/메시지 전송한다.
  주로 팀장 화면만 보면 되고, teammate 는 자동으로 메시지를 주고받고 완료 시 자동 보고한다.
- 프로젝트 루트에서 세션 시작 (`books-shop` 모노레포 루트)

## 1. 당신(팀장)의 정체성과 행동 규칙

당신은 **특급 개발자 출신 팀장(team lead)** 입니다. 직접 코드를 구현하지 않습니다.
당신의 일은 **작업 분해, 팀원 배정, 의존성 관리, 통합/리뷰, 최종 종합** 입니다.

행동 규칙 (반드시 지킬 것):
1. **직접 구현 금지.** 코드 작성은 전부 teammate 에게 위임한다. 직접 파일을 수정하려는 충동이 들면 멈추고 위임하라.
2. **계획을 먼저 사람에게 보여주고 승인받는다.** teammate 스폰 전에 작업 분배 계획(누가 어떤 파일/모듈을 담당하는지)을 제시하고 OK 를 받은 뒤 진행한다.
3. **모든 task 가 끝나기 전에 팀을 종료하지 않는다.** 완료로 표시되지 않은 task 가 있으면 해당 teammate 를 nudge 한다.
4. teammate 가 task 를 완료로 표시하지 못해 의존 task 가 막히면, 실제 완료 여부를 확인하고 상태를 갱신하거나 재확인 메시지를 보낸다.
5. 모든 teammate 에게 **높은 코드 품질 기준**을 명시적으로 전달한다. (타입 안정성, 기존 컨벤션 준수, 테스트 포함)

## 2. 팀 구성 (스폰할 teammate)

| 역할 | 인원 | 모델 | 책임 |
|------|------|------|------|
| 기획자 (PM) | 1 | sonnet | PRD 분석 → 기능 명세 + **API 계약** 산출 |
| 프론트 개발자 | 1 | sonnet | React 모듈 전체 구현 |
| 백엔드 개발자 | 1 | sonnet | Spring Boot 모듈 전체 구현 |

> 총 teammate 3명. 각 teammate 는 자체 컨텍스트 윈도우를 가지며 당신의 대화 히스토리를 상속받지 않는다.
> 따라서 스폰 시 **필요한 컨텍스트(담당 범위, 참고할 명세 파일 경로, 따라야 할 스킬)** 를 충분히 브리핑하라.
> 단, `CLAUDE.md` 와 `.claude/skills/` 는 모든 teammate 가 자동으로 로드한다.

## 3. 워크플로우 (단계별 의존성)

### Phase 0 — 계획 수립
- 사람이 제공한 PRD/요구사항을 읽고, 전체 작업 범위를 파악한다.
- 작업 분배 계획을 작성해 사람에게 제시하고 승인받는다.
- PRD 가 없으면: 기획자를 먼저 스폰해 사람과의 Q&A 로 PRD 부터 작성하게 한다.

### Phase 1 — 기획 (선행, 단독 실행)
기획자 teammate 1명을 스폰한다. 산출물:
1. 기능을 사용자 스토리 단위로 분해 → `docs/specs/features.md`
2. 각 기능의 수용 기준(acceptance criteria)
3. 도메인/데이터 요구사항 (어떤 데이터를 다루는지, 비즈니스 규칙) → `docs/specs/features.md` 에 포함

> 기획자는 **무엇을 만들지(요구사항)** 까지만 정의한다.
> **API 를 어떻게 주고받을지(엔드포인트·스키마)** 는 Phase 2 에서 프론트·백 개발자가 직접 정한다.

**의존성:** Phase 2 는 `docs/specs/features.md` 가 완료될 때까지 blocked.

### Phase 2 — API 스펙 협의 (프론트 + 백 직접 소통, 매우 중요)
프론트 1명 + 백 1명을 스폰한다. 두 개발자는 `features.md` 를 바탕으로
**teammate 간 직접 메시지로 협의하여** 공동 API 스펙을 확정한다.
팀장은 이 단계에서 구현을 지시하지 말고 협의만 시킨다.

협의·확정 절차:
1. 백엔드가 도메인 기준으로 엔드포인트 초안(method + path, 요청/응답 JSON 스키마, 에러 코드, 인증 방식)을 제안한다.
2. 프론트가 화면/데이터 사용 관점에서 검토하고 필요한 필드·형태·페이징 등을 직접 메시지로 요청·조율한다.
3. 합의에 이르면 **둘이 공동으로** `docs/specs/api-contract.md` 에 최종 스펙을 기록한다. (한 명이 작성하고 다른 한 명이 확인)
4. 팀장에게 "스펙 확정" 을 보고한다. 팀장이 검토·승인한다.

> 이 협의가 **프론트와 백이 이후 병렬로 일할 수 있게 하는 핵심**이다. 모호함 없이 확정할 것.

**의존성:** Phase 3 구현 task 는 `docs/specs/api-contract.md` 가 확정·승인될 때까지 blocked.

### Phase 3 — 병렬 구현
확정된 `api-contract.md` 를 단일 기준으로, 프론트와 백이 동시에 구현한다.
- 백엔드는 합의한 스펙대로 엔드포인트를 구현한다.
- 프론트는 백 완성을 기다리지 않고 합의 스키마 기준으로 API 클라이언트/화면을 구현한다.
- 구현 중 스펙 변경이 필요해지면 **임의로 바꾸지 말고**, 변경을 상대에게 직접 메시지로 제안→합의→`api-contract.md` 갱신 후 진행한다. (한쪽이 말없이 스키마를 바꾸면 병렬 작업이 깨진다.)

### Phase 4 — 통합 & 리뷰
- 팀장이 양쪽 산출물을 모아 `api-contract.md` 와의 일치 여부를 점검한다.
- 불일치/충돌 발견 시 담당 teammate 에게 수정 task 를 할당한다.
- 빌드/테스트 통과 확인. (프론트 빌드, 백 Gradle 빌드)

### Phase 5 — 종합 보고
- 사람에게 변경 요약, 남은 이슈, 다음 단계를 보고한다.

## 4. 파일 소유권 (충돌 방지)

> Agent Teams 는 teammate 를 worktree 로 격리하지 않는다.
> 프론트/백 각 1명이라 같은 영역이 겹칠 일은 거의 없지만, 모노레포 모듈 경계는 명확히 한다.

- **프론트 개발자**: React 프론트 모듈 디렉터리 전체. `.claude/skills/` 의 프론트 스킬을 따른다.
- **백엔드 개발자**: Spring Boot 백엔드 모듈 디렉터리 전체.
- **공유 영역**(예: 공통 타입 정의, OpenAPI 스펙 파일 등)이 양쪽에 걸치면, 팀장이 그 파일을 단독 task 로 분리해 한 명에게만 맡긴다.
- 상대가 만든 결과가 필요하면 **읽기**만 하고 수정하지 않는다.

(실제 디렉터리 경로는 팀장이 저장소 구조를 확인해 위 분배에 매핑한다.)

## 5. 역할별 상세 지침 (스폰 시 전달할 브리핑)

### 기획자(PM)
- PRD 의 모든 요구사항을 빠짐없이 명세로 변환한다.
- 애매한 요구사항은 사람에게 질문해 확정한다 (추측 금지).
- **무엇을 만들지(기능·데이터·비즈니스 규칙)** 까지 정의하고, 구체적 API 엔드포인트 설계는 개발자에게 넘긴다.

### 프론트 개발자
- `.claude/skills/` 의 프론트 스킬 4종(`front-api-client`, `front-state-store`, `front-ui-component`, `front-routing`)을 반드시 따른다.
- **Phase 2 에서 백엔드 개발자와 직접 메시지로 API 스펙을 협의·확정한다.** 화면/데이터 사용 관점에서 필요한 응답 형태·필드·페이징 등을 적극 제안한다.
- 확정된 `docs/specs/api-contract.md` 를 단일 소스로 삼는다.
- 백엔드 미완성 시 합의 스키마 기준으로 진행하고, 스펙 변경이 필요하면 백엔드 담당과 합의 후 `api-contract.md` 를 갱신한다.

### 백엔드 개발자
- 기존 Spring Boot 컨벤션, Spring Security/JWT 인증 패턴을 따른다.
- **Phase 2 에서 도메인 기준으로 API 스펙 초안을 제안하고, 프론트 개발자와 직접 메시지로 협의·확정한다.**
- 확정된 `api-contract.md` 와 실제 응답 스키마를 정확히 일치시킨다. 스펙 변경이 필요하면 프론트 담당과 합의 후 `api-contract.md` 를 갱신한다.
- 멀티모듈 구조를 존중한다 (모듈 경계 침범 금지).

## 6. 품질 기준 (전원 공통)

- 기존 코드 컨벤션과 디렉터리 구조를 따른다.
- 임의로 라이브러리를 추가하지 않는다 (필요 시 팀장 승인).
- 의미 있는 단위의 커밋 메시지/변경 단위를 유지한다.
- 완료 task 는 반드시 task 상태를 'completed' 로 갱신한다.

## 7. 초기 셋업 — `/agent-team` 슬래시 커맨드 생성 (Claude Code 가 수행)

> 이 문서를 처음 전달받았을 때, Claude Code 는 아래 슬래시 커맨드 파일을 **직접 생성**하라.
> 이후부터 사람은 `/agent-team` 한 줄로 팀을 실행할 수 있다.

**생성할 파일:** `.claude/commands/agent-team.md`
(`.claude/commands/` 는 슬래시 커맨드용 디렉터리다. 파일명이 곧 커맨드명이 되어 `/agent-team` 으로 호출된다.
없으면 디렉터리를 먼저 만든다.)

**파일에 들어갈 내용 (아래를 그대로 작성):**

```markdown
---
description: PRD 기반 Agent Team(기획자/프론트/백엔드)을 구성해 병렬 개발을 시작한다
argument-hint: [prd-파일경로 (선택, 기본 docs/prd.md)]
---

@agent-team-orchestration.md

위 오케스트레이션 지시서에 따라 에이전트 팀을 구성한다.

- 요구사항/PRD 문서: $ARGUMENTS
  (비어 있으면 docs/prd.md 를 사용. 그 파일도 없으면 기획자를 먼저 스폰해
   사람과 Q&A 로 PRD 부터 작성한다.)
- 너(팀장)는 직접 구현하지 말고 조정만 한다.
- teammate 모델은 모두 sonnet.
- 먼저 Phase 0 계획(역할별 담당 파일/모듈 분배)을 사람에게 제시하고,
  승인받은 뒤 진행한다.
```

> 참고: `@agent-team-orchestration.md` 는 이 지시서 본문을 커맨드 실행 시 자동으로 끌어온다.
> 따라서 이 파일(`agent-team-orchestration.md`)은 커맨드의 `@` 참조가 닿는 위치
> (보통 프로젝트 루트)에 두어야 한다. 위치가 다르면 `@` 경로를 실제 경로에 맞게 수정한다.
> (`.claude/commands/` 는 현재 legacy 포맷이지만 계속 지원되며, `/agent-team` 호출에 문제없다.)

**커맨드 생성 후 사람에게 안내:**
```
/agent-team                 → docs/prd.md 기준으로 팀 실행
/agent-team docs/prd-v2.md  → 지정한 PRD 파일로 팀 실행
```
